Note on Different Court Policies

The Journal Technologies case management system (CMS) allows each California Court to configure their system to precisely meet their requirements. This means there can be considerable differences between Courts in areas such as code lists, document metadata, workflows, etc.

You must assume that EACH Court will have different Court Policy values from each other.
For example, Civil Limited case type code for Los Angeles Superior Court is LC while it's 421110 for Placer Superior Court.

Similarly, not only are the document list codes in Placer different than Los Angeles, the case category associations for those documents may be different.

Code List Relationships

Code lists are used to constrain the allowable values for certain information in an ECF 4.0 message. Journal Technologies EFM has extended the Genericode standard to include relationship information with other code lists. For example, the end-user would first select the case type (ex. Civil Limited) in order to load the correct case categories (ex. Asbestos Property Damage).

  • case categories within a case type
  • party roles within a case category
  • document types within a case type
  • etc

Basic Structure

Relationships are mapped using a custom extension called <ns4:relationships>. Here’s the basic structure: 

<ns2:CodeList xmlns:ns2="http://docs.oasis-open.org/codelist/ns/genericode/1.0/" 
xmlns:ns3="urn:com.journaltech:ecourt:ecf:extension:DocumentValue" 
xmlns:ns4="urn:com.journaltech:ecourt:ecf:extension:CodeValue">
<SimpleCodeList>
   <Row>
      <Value ColumnRef="code">
         <SimpleValue><CODE></SimpleValue>
      </Value>
      <Value ColumnRef="value">
      <ComplexValue xmlns:xsi="http://www.w3.org/2001/XMLSchema-instance" xsi:type="ns4:CodeValue">
         <ns4:name>VALUE</ns4:name>
         <ns4:relationships>
            <ns4:relationship>
                <ns4:relatedListName>RELATED_CODE_LIST</ns4:relatedListName>
                 <ns4:relatedCode>RELATED_CODE_LIST</ns4:relatedCode>
         </ns4:relationship>
         </ns4:relationships>
      </ComplexValue>
      </Value>
   </Row>
</SimpleCodeList>
</ns2:CodeList>

Main Points

  • This basic pattern will repeat across all our code lists.
  • EFSPs can construct pre-built relationships between things like dropdown lists. For example, the appropriate categories can be displayed in a dropdown only after the user selects the case type.

Lookup: Attorneys

Journal Tech EFM provides an optional attorney lookup endpoint.

/ws/rest/ecourt/niem/attorneyList

Available query parameters:

  • barNumber (exact match)
  • firmName (contains)
  • firstName (starts with)
  • lastName (exact match)
Examples:

By Bar Number:
/ws/rest/ecourt/niem/attorneyList/barNumber/BARNUMBER

By Firm Name:
/ws/rest/ecourt/niem/attorneyList/firmName/FIRMNAME

By First Name:
/ws/rest/ecourt/niem/attorneyList/firstName/FIRSTNAME

By First Name + Last Name:
/ws/rest/ecourt/niem/attorneyList/firstName/FIRSTNAME/lastName/LASTNAME

Lookup: Court Location

Journal Tech EFM provides a zip code lookup for matching the Incident Zip Code with the correct Courthouse code.

/ws/rest/ecourt/niem/courtLocations/zipCode

Available query parameters:

  • zipCode (exact match)
  • caseType (code)
  • caseCategory (code)
Examples Requests:

By ZipCode:
/ws/rest/ecourt/niem/courtLocations/zipCode/90012

By ZipCode + CaseType:
/ws/rest/ecourt/niem/courtLocations/zipCode/90012/caseType/CU

By ZipCode + CaseType + Case Category:
/ws/rest/ecourt/niem/courtLocations/zipCode/90012/caseType/CU/caseCategory/0201

Examples Response:
