Notify Filing Review Complete

Overview

Journal Tech returns multiple NFRCs back to the original webhook defined in your filing. These NFRCs can occur at different times and reflect the various court processing statuses a filing will move through:

NFRC Multiple Steps
NFRC #1 - "Clerk Action"
Returned immediately after Clerk takes an action (accept or reject). This includes the filing and document filing statuses, conformed copies and any additional documents such as:
  • Notice of E-Filing Confirmation
  • Notice of Case Assignment
  • Etc
NFRC #2 - "Financials"
Generating a copy of the actual cms receipt is processor intensive so JTI EFM performs this step separately. This second NFRC can be returned anywhere from a few seconds after NFRC 1 or (due to load) several hours. It will contain the exact contents of NFRC 1 plus:
  • Copy of CMS receipt
  • Breakdown of fees
NFRC #3 - "Delayed"
These are typically generated and returned to EFSPs after a judicial officers have rendered some type of decision. It will contain the exact contents of NFRC 2 plus any documents. Typical examples would be pleading papers with "Proposed Order"such as:
  • Application and (Proposed) Order
  • Judgments
  • Orders
  • Stipulation and Orders

NOTE: Not all Courts support pleading papers through eFiling. Refer to Subsequent Filing concepts for Court-specific configurations.

NFRC Explanations

This sample showcases nearly all elements and structure you'll encounter in an NFRC.

Request Parameter Description
ReviewFilingCallbackMessageExt This extension allows us to track all filing identifiers supplied by each major design element (MDE).
  • FILING_ASSEMBLY_MDE = This is your own unique filing identifier that you passed during case initiation. This identifier is mainly used by EFM for debugging purposes and auditing.
  • FILING_REVIEW_MDE = The DocumentFileControlID returned by the EFM and returned during ReviewFilingCallbackMessage.
  • COURT_RECORD_MDE = If available, its the filingId generated by the CMS. Typically this field is blank.
FilingStatusCode This parameter under <FilingStatus> which signifies the overall filing status as determined by the Court.
  • "RECEIVED_UNDER_REVIEW" = filing was received and is under review by the court
  • "ACCEPTED" = filing was accepted
  • "PARTIALLY_ACCEPTED" = filing was received, some documents were accepted, and some documents were rejected
  • "REJECTED" = filing was received and completely rejected
DocumentFilingStatusCode This parameter under <DocumentFilingStatus> which signifies the overall filing status of the document.
  • "RECEIVED_UNDER_REVIEW" = document was received and is under review by the Court
  • "RECEIVED" = document was received
  • "ACCEPTED" = document was accepted
  • "REJECTED" = document was rejected
DocumentStatus The elements under this node indicates document status in the actual cms. The <DateRepresentation> indicates the timestamp of the status in the cms. The <StatusText> (if available) indicates the status:
  • "R" = Received
  • "F" = Filed
  • "I" = Issued
  • "RJ" = Rejected
  • "RP" = Proposed - Received
  • "FG" = Filed and Granted
DocumentDispositionType The elements under this node indicates the document disposition status. This usually pertains to documents in NFRC #3 that required judicial intervention. <DateRepresentation> indicates the timestamp of the status in the cms. The <DocumentDispositionType> indicates the ruling status such as:
  • "GRA" = Granted
  • "DEN" = Denied
  • "ORD" = Ordered
  • "OAI" = Ordered and Issued
BinaryLocationURI The URI where the document can be downloaded from. By default, links last for 7 days after initial NFRC generation.

Rejected and Partially Rejected Filings

If your FilingStatus is REJECTED, the clerk has rejected your entire filing. Since you won't receive any conformed copies, the end user will simply need to read the reasons for rejection, fix the issues and then start a completely new filing.

If your FilingStatus is PARTIALLY_ACCEPTED, the filing was received, some documents were accepted, and some documents were rejected. A case WAS created (the user will get a caseNumber) but additional documents are expected to be filed on that case. The end user should first download the accepted documents (which are conformed copies). Then for the rejected documents, the user will need to read the reject reasons on the document, fix the issues and then start a subsequent filing. The user does NOT need to include the previously accepted documents in this filing. For example, if the Summons gets rejected from the case initiation, the Summons must be submitted as a subsequent filing to the case number that was generated.
