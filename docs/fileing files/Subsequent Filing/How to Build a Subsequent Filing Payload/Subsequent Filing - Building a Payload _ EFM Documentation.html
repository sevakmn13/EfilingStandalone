Subsequent Filing - Building a Payload

This document walks through the construction of a general ReviewFiling payload for subsequent filing.

STEP 1: GetCase

For all subsequent filings, the first step is to call GetCase to retrieve the case information. In the Case Model, you can see that cases can have one or more complaints ("sub-cases"). These sub-cases represent the original complaint, cross-complaint, counter claim, etc. Each sub-case has it’s own Id and each party has their own Id. Here are two examples:

Request Parameter Description
IdentificationID

(SendingMDELocationID)

 URL endpoint on your server where EFM will return all responses to for the lifecycle of this filing.
IdentificationID

(DocumentIdentification )

 This is your own unique filing identifier that is passed back during NotifyFilingReviewComplete. This identifier is mainly used by EFM for debugging purposes and auditing.
CaseDocketID An identifier used to reference a case on a docket. This is often known as the "case number". It's a publicly meaningful identifier usually displayed on public court calendars".
Complaint In the LASC case model, each case is made up of one or more complaints. Each complaint has it’s own Id and each party (not person) has their own Id. You set the complaint Id in the id attribute of this <Complaint> element to indicate which complaint the document needs to be filed on. In this example the complaint Id is 4000954

When multiple sub-cases exist on the case (i.e. original complaint, cross-complaint, counter claim, etc), you will need to prompt the user to select which complaint they are filing the document on. If there is only one complaint, pass the first and only complaint Id with the filing. The filing will fail immediately if this flag is set to true and you do not send the complaint Id .

Request Parameter Description
FilingLeadDocument The lead document in the filing sequence. JTI EFM only accepts one lead document per filing.
FilingConnectedDocument Any additional documents attached to the filing
BinaryFormatStandardName The mime type of the document. Currently LASC only accepts application/pdf and mime types associated with MS word (ex. application/msword and application/vnd.openxmlformats-officedocument.wordprocessingml.document
BinaryLocationURI EFM will download the document from this location. It's the EFSPs responsibility to ensure the document is available.
DocumentDescriptionText The document code. This would likely be what "document" the end-user is filing. The document code value is defined in DocumentIdentification under Court Policy.
DocumentFileControlID Optional. Your own identifier for this document.
DocumentSequenceID The sequence number of this document in a filing.
documentFilingMetaDataItem The document metadata (or information) that needs to be collected about each document as defined by DocumentIdentification in Court Policy. In this example, the document being filed is a Answer ("RES010") and contains three metadata items.

Filed By
The first metadata item is the existing party (classType="caseParticipant") who is filing the document (FILED_BY). We signify existing data using the <valueRestriction> element with the value "existing-data". The end user selected this filed by party from a list of parties on the case retrieved during GetCase. The <idReferences> contains the partyId of "filed by" party.

New Party
The second metadata item indicates a new party (classType="caseParticipant") is being added as a filing party (NEW_FILED_BY). We signify this is a new record using the <valueRestriction> element with the value "new-data". The end-user as input all the relevant party information along with AKA information.

New Attorney
The third metadata item indicates a new attorney (classType="attorney") is being added as a filing party (FILED_BY_REPRESENTATION). We signify this is a new record using the <valueRestriction> element with the value "new-data". The end-user as input all the relevant attorney information.
Filed By vs. New Party
LASC will allow the majority of subsequent filings to be submitted with either an Existing Party OR a New Party. However, the following documents will require submitting an Existing Party in order to add a New Party:
  • NTC020 Notice of Add-on to JCCP Petition
  • NTC021 Notice of Add-on to JCCP Petition (short form)
  • COM070 Cross-Complaint
  • ISS030 Summons (with Amended Complaint, Amended Cross Complaint, Cross-Complaint)
The list of Party Types end-users can choose from are defined in Document Metadata as partyTypes under the NEW_FILED_BY or NEW_AS_TO codes. For example, the document Summons Joint Debtor (ISS040) allows for a NEW_FILED_BY to be created with party types like Appellant, Respondent, Cross-Appellant, Witness, etc. Disregard the participant filters. If no partyTypes are displayed, it's a misconfiguration and you should enter an EFSPHD ticket.
messageToClerk Any additional message the end user wishes to communicate to the Court about the filing.
