Are there any issues with calling your SOAP services with .NET C# WCF?

No. The JTI solution is independent of any particular technology stack. As long as standard ECF 4.0 is used to communicate using SOAP and REST APIs, you will be fine with whatever technology stack that is desired. Back to top

Is JTI supporting a REST API (in addition the existing SOAP)?

JTI is currently using REST API endpoints for code lists (see GetPolicy). The code lists return NIEM xml. Once we have completed the standard ECF calls, a more aggressive REST API solution for additional eFiling calls will be made available. Back to top

Has JTI extended the OASIS ECF 4.01 with their own namespaces/data?

Yes. A good way to see the JTI extensions is to open the GetCase request in Boomerang and search the namespaces for "journaltech".

Here are some extensions that have been made:
  • DocumentValue - needed to support our code list hierarchies (i.e., which case categories relate to which case types)
  • DocumentFilingMetaData - needed to support document metadata
  • CivilCaseTypeExt - needed to support the concept of SubCase/Petition/Cross-complaints
These extensions will be documented on this site very soon. Back to top

EFSP IP Addresses - will they need to be white listed for EFM access?

JTI will be using the combination of a basic authentication + IP whitelisting for production. These authentication credentials will be different than your current testing/sandbox server credentials. Individual EFSPs will be issued their production credentials as the go-live dates approach. Back to top

Are EFSPs expected to perform a virus scan on documents before submitting?

Yes, EFSPs are expected to provide an initial level of protection and perform a virus scan on all documents before submitting. If a virus is detected by the EFSP, the submission should not be sent. If a virus avoids the EFSP scan but is detected by the downstream EFM, the filing will be rejected during NotifyFilingReviewComplete callback. The following is a contract excerpt which indicates the EFSP does need to scan prior to sending a document:

2.4. Certain Specific Limitations. The EFSP shall not, and the EFSP shall not permit any person (including, without limitation, employees, contractors, subcontractors, consultants, agents and Electronic Filers) to (a) copy or otherwise reproduce, reverse engineer or decompile all or any part of the JTI E-Filing Application or E-Filing System, (b) make alternations to or modify the JTI E-Filing Application or E-Filing System, (c) grant sublicenses, leases or other rights in or to the JTI E-Filing Application or E-Filing System without the prior written consent of JTI, (d) propagate any virus, worms, Trojan horses, or other programming routine intended to damage the JTI E-Filing Application or E-Filing System or any other system or data or (e) or permit any party access to the JTI E-Filing Application or E-Filing System for purposes of programming against it.

How is JTI going to handle User Management and Authentication?

Our EFM authenticates your application, not individual users. Therefore, you're free to manage your own users (LDAP, etc). Back to top

How do we handle rejected and partially rejected filings?

If your FilingStatus is REJECTED, the clerk has rejected your entire filing. Since you won't receive any conformed copies, the end user will simply need to read the reasons for rejection, fix the issues and then start a completely new filing.

If your FilingStatus is PARTIALLY_ACCEPTED, the filing was received, some documents were accepted, and some documents were rejected. A case WAS created (the user will get a caseNumber) but additional documents are expected to be filed on that case. The end user should first download the accepted documents (which are conformed copies). Then for the rejected documents, the user will need to read the reject reasons on the document, fix the issues and then start a subsequent filing. The user does NOT need to include the previously accepted documents in this filing. For example, if the Summons gets rejected from the case initiation, the Summons must be submitted as a subsequent filing to the case number that was generated.

Please see the list of filing status codes in our NFRC documentation. Back to top
