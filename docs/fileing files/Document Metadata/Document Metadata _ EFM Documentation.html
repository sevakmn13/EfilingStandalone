Overview

The JTI EFM requires EFSPs to supply metadata information with every filing. The metadata for each document is defined by the Court through the DocumentIdentification endpoint of Court Policy. The information is typically entered by the end user so it's the responsibility of the EFSP is to:
  1. interpret the metdata defined for a particular document
  2. render the appropriate form to the end user based on the metadata item
  3. include this user supplied metadata information in the filing

Core Structure

Each DocumentIdentification listed in Court Policy follows this general structure: 

<Row>
   <Value ColumnRef="code">
       <SimpleValue>DOCUMENT_CODE</SimpleValue>
   </Value>
   <Value ColumnRef="value">
       <ComplexValue xmlns:xsi="http://www.w3.org/2001/XMLSchema-instance" xsi:type="ns4:DocumentValue">
           <ns4:efmRequiresSubCase>BOOLEAN</ns4:efmRequiresSubCase>
           <ns4:name>NAME_OF_DOCUMENT</ns4:name>
           <ns4:metadata>
               <ns4:metadataItems>
                   <ns4:code>CODE</ns4:code>
                   <ns4:name>NAME</ns4:name>
                   <ns4:description>DESCRIPTION</ns4:description>
                   <ns4:required>REQUIRED</ns4:required>
                   <ns4:multiple>MULTIPLE</ns4:multiple>
                   <ns4:classType>CLASSTYPE</ns4:classType>
                   <ns4:valueRestriction>VALUE_RESTRICTION</ns4:valueRestriction>
                   <ns4:additionalInfoTags>ADDITIONAL_INFO_TAGS</ns4:additionalInfoTags>
               </ns4:metadataItems>
           </ns4:metadata>
           <ns4:caseTypes>
               <Row>
                   <Value ColumnRef="code">
                       <SimpleValue>CASE_TYPE</SimpleValue>
                   </Value>
               </Row>
           </ns4:caseTypes>
           <ns4:caseCategories>
               <Row>
                   <Value ColumnRef="code">
                       <SimpleValue>CASE_CATEGORY</SimpleValue>
                   </Value>
               </Row>
           </ns4:caseCategories>
           <ns4:formGroups>
               <Row>
                   <Value ColumnRef="code">
                       <SimpleValue>FORM_GROUP</SimpleValue>
                   </Value>
               </Row>
           </ns4:formGroups>
       </ComplexValue>
   </Value>
</Row>

Request Parameter Description
SimpleValue String. This is the document code as defined by the Court.
efmRequiresSubCase Boolean. If true, you MUST indicate which complaint the document needs to be filed on. When multiple complaints exist on the case, you will need to prompt the user to select which complaint they are filing the document on. If there is only one complaint, pass the first and only complaintId with the filing. The filing will fail immediately if this flag is set to true and you do not send the subCaseId.

This concept is critical for subsequent filing. You can also view the case model for more information about complaints.
<metadataItems>
    NAME 		String. The name of the document as defined by the Court. This is typically the name that is presented to the user in a dropdown list of available documents.
<metadataItems>
    CODE 		String. Codes provide you with the "intent" of the metadata item. See Code Types for information.
<metadataItems>
    DESCRIPTION 		String. The description of the metadata item.
<metadataItems>
    REQUIRED 		Boolean. Defines whether the metadata item is required or optional information.
<metadataItems>
    MULTIPLE 		Boolean. Whether this metadata item can be repeated. How you render form information depends on the valueRestriction and classType.

For example, if <multiple> is "true" during subsequent filing when <valueRestriction> is "existing-data" and <classType> is"caseParticipant", it means you can display checkboxes next to each party in the case when selecting a Filed By party. However, if <multiple> is "false" in this scenario, it means a radio button is to be used next to each party.
<metadataItems>
    CLASSTYPE 		String. Essentially the data type required for the metadata item. This can be anything from a simple currency or date to a standard object such as an Attorney. See Class Types for information.
<metadataItems>
    FILTER 		String. Indicates what filters should be applied to the data being displayed. How you render form information when a filter is being applied depends on the valueRestriction and classType. See Filter Types for a list of filters.

For example, if <filter> is "caseParticipant-all-parties-except-attorney" during subsequent filing when <valueRestriction> is "existing-data" and <classType> is"caseParticipant", it means you should filter out attorneys from the list of parties being displayed. However, if <filter> is blank , it means no filters need to be applied.
<metadataItems>
    VALUE_RESTRICTION 		String. The possible values are "existing-data" and "new-data". If existing-data, it means the record for this metadata item should be pulled from existing case data retrieved from GetCase. If new-data, it means the user is able to create a new record.
<metadataItems>
    ADDITIONAL_INFO_TAGS 		Boolean. These are simply boolean "flags" that should be rendered as simple checkboxes or yes/no radio buttons. Typically these would ask the user yes/no questions such as (i) are you a self-represented litigant, (ii) does this party require a fee waiver, (iii) should this case be conditionally sealed, etc.
<caseTypes>
    CASE_TYPE 		String. The case type(s) which this document can be filed.
<caseCategories>
    CASE_CATEGORY 		String. The case category(s) which this document can be filed.
<formGroups>
    FORM_GROUP 		String. The form groups which this document can be filed.

Case Initiation:
Lead Documents:
  • Only documents that contain formGroup EFCI_LEAD

Additional Documents:
  • Only documents that contain formGroup EFCI

NOTE: For Case Initiation, if a document DOES NOT contain a formGroup it should NOT be used for case initiation
Subsequent Filing:
Lead Documents:
  • Only documents that contain formGroup EF_LEAD
Additional Documents:
  • All documents in the Court Policy Document List are available here.

Code Types

These provide you with the "intent" of the metadata item. The code types below have specific meanings.

Request Parameter Description
FILED_BY Indicates the selected case participant(s) are filing the documents.
AS_TO Indicates the document is being filed "as to" the selected case participant(s).
NEW_FILED_BY Used when the metadata item is a new "filed by" case participant that does not yet exist on the case.
JUDGMENT The metadata item is a judgment of some kind.
EXISTING_ATTORNEY The metadata item is an existing attorney on the case.
FILED_BY_REPRESENTATION Indicates the representation record of the "filed by" party.
REQUESTED_BY Indicates the party is requesting the action (ex. Request for Dismissal)
DISMISSAL_PREJUDICE_TYPE Type of dismissal, typically with or without prejudice.
NEW_AS_TO Used when the metadata item is a new "as to" case participant that does not yet exist on the case.
DEFERRED_PAYMENT Used to give end-user option to deferred paying filing fees.

Contact Information & Self Represented

JTI does not require an extra data element that designates a party as self-represented. The EFM designates a party or parties is self-represented if there is no attorney associated in the XML. This applies for both Person and Organization Entity types. If attorney contact information is not provided then the party is deemed to self-represented and contact information, including email and mailing address, must be provided.

  • MAILING Address Type is required for all addresses
  • WORK Telephone Type is required for telephone number. Recommended format 9999999999 (no hyphens)
  • EMAIL Addresses are required for all attorneys and/or Self Represented Litigants and validated to standard local-part@domain,com format.

Contact name requirements are as follows:

Contact Name Requirement
First Name Required
Middle Name/Initial Optional
Last/Sur Name Required
Law Firm Name Required if No Attorney Name, otherwise optional
Attorney First Name Required
Attorney Middle Name Optional
Attorney Last Name Required

Filters

Property indicated how you should filter the case data being displayed to the user. Typically used when displaying existing parties on a case.

Request Parameter Description
case_participant_all_parties_except_attorney Exclude attorneys when displaying parties on the case.
case_participant_plaintiff Only display plaintiffs.
case_participant_defendant Only display defendants.
case_participant_attorney Only display attorneys.

Association Types

Indicates the type of associations (i) between participants and (ii) participants with documents when mapping relationships during Case Initiation.

Request Parameter Description
REPRESENTEDBY Relationship between a party and their attorney.
FILEDBY Relationship between a party doing the filing and a document.
REFERS_TO Relationship between a document being filed and the "as to" party.
