BinaryLocationURI

Document Naming
Our XML parser expects a well-formed URI. Use DNS and web-safe character sets for document names:
  • Alphanumeric characters: 0-9 a-z A-Z
  • Special characters: ! - _ . * ' ( )
  • No Spaces!

No Spaces
You MUST remove all spaces from the entire value in BinaryLocationURI including document name. Our XML parser expects a well-formed URI and will fail if the URI contains spaces. Clerks will typically reject filings if the JTI EFM cannot retrieve the documents from your storage location.

URL Encoding
JTI automatically runs urlDecode() on the BinaryLocationURI value you send. Please BE AWARE of how your URLs are generated by your storage provider. For example, storage providers like AWS S3 return their URLs already encoded so there's no need for your application to encode it a second time.

Document Name Extension
You MUST provide a name extension (.pdf) on your uploaded documents. The EFM (by design) doesn't auto-fix these types of issues. Since the stamping functionality will fail on documents without name extensions, there is a high chance Clerks will reject the entire filing.

Address Country Codes

Address Country Code must be passed wherever you're adding a new address (ex. party address, attorney address) during ReviewFiling. Country codes can be found in Court Policy with type ADDRESS_COUNTRY.

Validation Rules
  • If country code is the United States, then Address 1, City, State and Zip Code are mandatory fields.
  • If country code is NOT the United States, then Address 1, City are mandatory while Zip would be optional

When Enforced
There are two implementations where these country code validation rules will be enforced.

1. StructuredAddressType
Whenever your metadata class type requires StructuredAddressType, the XML tag name for country code is LocationCountryName. This means:
  • Case Participant
  • Attorney
  • Address

The following code example is a sample attorney address from Uruguay being added during case initiation: 

<ci1:ContactInformation xmlns:ci1="http://niem.gov/niem/niem-core/2.0">
    <ci1:ContactMailingAddress>
      <sta1:StructuredAddress xmlns:sta1="urn:com.journaltech:ecourt:ecf:extension:StructuredAddressExt">
        <sta1:LocationType>M</sta1:LocationType>
        <ci1:LocationCountryName>UY</ci1:LocationCountryName>
        <ci1:AddressDeliveryPoint xsi:type="ci1:StreetType">
          <ci1:StreetFullText>18 de Julio Avenue</ci1:StreetFullText>
        </ci1:AddressDeliveryPoint>
        <ci1:LocationCityName>Montevideo</ci1:LocationCityName>
        <ci1:LocationPostalCode>11200</ci1:LocationPostalCode>
      </sta1:StructuredAddress>
    </ci1:ContactMailingAddress>
    <ci1:ContactEmailID>person@aol.com</ci1:ContactEmailID>
</ci1:ContactInformation>

2. Contact
If you are doing a subsequent filing and your metadata class type is Contact, the XML tag name for country code is simply country. The validation rules will be enforced whenever a new Contact object is supplied when submitting document types of form group FILED_BY_NEW. For example:
  • Answer
  • Amended Complaint
  • Motion to Dismiss
  • Response to Petition
  • etc

The following is a sample country code for Contact metadata item when including a new party address for an Answer: 

<ns9:contactValue>
    <ns10:address1>12345 Wilshire BLvd</ns10:address1>
    <ns10:city>Long Beach</ns10:city>
    <ns10:zip>90012</ns10:zip>
    <ns10:state>CA</ns10:state>
    <ns10:country>US</ns10:country>
    <ns10:telephoneType>W</ns10:telephoneType>
    <ns10:email>book@journaltech.com</ns10:email>
</ns9:contactValue>

Party Sub-Type

Some Courts (ex. LASC) requires end-users to select any applicable "party sub-type" whenever adding a new party in a filing. Party sub-types are boolean attributes about a new party. For example, this could mean the party is a minor, a guardian ad litem or deemed incompetent per CA §372 and §373.

The url for retrieving the list of party sub-types can be found in Court Policy under the PARTY_SUBTYPE. Here's an XML snippet for including Minor ("MIN") and Guardian Ad Litem ("GAL") party sub-types in ReviewFiling. 

<p:CaseParticipantExt xmlns:p="urn:com.journaltech:ecourt:ecf:extension:CaseParticipantExt" xmlns:st="http://niem.gov/niem/structures/2.0" st:id="filedBy0">
  <p1:CaseParticipantRoleCode xmlns:p1="urn:oasis:names:tc:legalxml-courtfiling:schema:xsd:CommonTypes-4.0">PLAIN</p1:CaseParticipantRoleCode>
  <p:partySubType>GAL</p:partySubType>
  <p:partySubType>IP</p:partySubType>
  <p2:EntityPerson xmlns:p2="http://niem.gov/niem/niem-core/2.0" xmlns:ps1="urn:oasis:names:tc:legalxml-courtfiling:schema:xsd:CommonTypes-4.0" xsi:type="ps1:PersonType">
    <p2:PersonName>
      <p2:PersonGivenName>Jeff</p2:PersonGivenName>
      <p2:PersonSurName>Mason</p2:PersonSurName>
    </p2:PersonName>
  </p2:EntityPerson>
</p:CaseParticipantExt>

To simplify the implementation for vendors, Journal Tech recommends rendering the sub-types as checkboxes in the UI (see below).

Auto Accept Test Filings

This is not for Certification Filings! Only test filings within UAT environments

 This process is to expedite transactions within UAT environments. This will enable test filings to be accepted without JTI manually reviewing the submission, resulting in a faster return on your end. Filings will still undergo normal verifications but data entered will not be reviewed. Certification filings will continue to have the manual approach.
Available Environments
LASC UAT
Riverside UAT
Placer UAT
Alameda UAT
Tulare UAT

Test Filings

When submitting a filing to the UAT environment, to have the filing automatically processed, input into the XML header this string "This is a test filing". As long as all validations are met, this will trigger the process within the UAT EFM which will then accept the filing and stamp documents.

Example XML

    <S:Header>
        <status xmlns="com.journaltech.niem.test">This is a test filing</status>
    </S:Header>

Auto Reject Test Filings

Only test filings within UAT environments

 This process is to expedite transactions within UAT environments. This will enable test filings to be rejected without JTI manually reviewing the submission, resulting in a faster return on your end. Data entered will not be reviewed, but will have the status of rejected. NFRC will be returned.
Available Environments
LASC UAT
Placer UAT
Tulare UAT

Test Filings

When submitting a filing to the UAT environment, to have the filing automatically rejected, input into the XML header this string "Auto Reject Filing". This will trigger the rejection process within the UAT EFM.

Example XML

    <S:Header>
        <status xmlns="com.journaltech.niem.test">Auto Reject Filing</status>
    </S:Header>
